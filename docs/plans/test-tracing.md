# End-to-End Tracing Plan for Banff E2E Test

## Goal

Enable end-to-end tracing for the Banff e2e test in [tests/e2e/banff/suites.go](/home/maru/src/avalanchego_maru-test-tracing/tests/e2e/banff/suites.go) as a proof of concept.

The implementation should proceed in phases rather than trying to solve node, client, and test instrumentation all at once.

The initial priority is not Banff-specific instrumentation. The initial priority is to make `--enable-tracing` configure a tmpnet-managed network so that AvalancheGo nodes emit traces, those traces are collected, and the results are queryable in Tempo and viewable in Grafana.

Only after that infrastructure is in place and validated should the work expand to client-side instrumentation and full test-level correlation.

The final proof of concept should provide:

- Total test duration
- Duration of each client API call
- Duration of relevant server-side request handling
- Correlation between client and server spans for the same request
- Enough output to compare where time is spent relative to the whole test

This plan is intentionally scoped to the Banff test only for the final proof of concept. The infrastructure phases should still be implemented at the tmpnet and e2e harness levels so they can be reused by later tests once the concept is validated.

## Success Criteria

This work has three distinct success checkpoints.

### Phase 1 success: local observability stack is manageable and verifiable

Phase 1 has two sub-phases. Phase 1a establishes a complete local monitoring stack for metrics and logs, validating the infrastructure pattern before adding tracing. Phase 1b extends that stack with Tempo for traces.

#### Phase 1a success: local metrics and logs are viewable in local Grafana

- A single flag `--start-local-monitoring-stack` (or equivalent `tmpnetctl` command) starts all required local services: Grafana, Prometheus (server mode), Loki, and the existing Promtail/Prometheus collectors reconfigured for local backends
- This is additive — the existing remote collection workflow (`--start-metrics-collector`, `--start-logs-collector` with `PROMETHEUS_*` / `LOKI_*` env vars) continues to work unchanged
- When running locally, collectors push to local Prometheus and Loki instead of requiring remote backend credentials
- Local Grafana is provisioned with Prometheus and Loki datasources pointing at the local backends, and with dashboards from the `ava-labs/avalanche-monitoring` repo
- An e2e test run with `--start-local-monitoring-stack` produces test output with Grafana links that open in local Grafana and show real data from the test run
- Stack startup includes readiness verification for all services
- The workflow is repeatable: start, stop, restart all work cleanly

#### Phase 1b success: local Tempo is added to the stack

- Tempo is added as another service in the local monitoring stack
- Grafana is provisioned with a Tempo datasource in addition to Prometheus and Loki
- The stack can be started with or without Tempo (Tempo is only needed when tracing is enabled)
- This sub-phase is a prerequisite for Phase 2 but does not require any node tracing configuration — it only needs the backend to be running and queryable

### Phase 2 success: tmpnet-managed node tracing works

- `--enable-tracing` configures the tmpnet-managed network so AvalancheGo nodes export traces
- Trace collection can be started and checked through tmpnet/e2e harness controls
- Traces emitted by nodes are collected and queryable programmatically
- A Grafana link can be emitted for the current network and opened successfully
- Trace visibility is confirmed both through automated checks and through manual GUI inspection
- Phase 2 can be considered complete even if there are not yet any client or test spans, so long as node-generated spans are visible and queryable

### Phase 3 success: Banff is instrumented end to end

- A root span exists for the test run
- Child spans exist for each logical test step or at least each RPC call
- Client spans exist for requests such as `avm.issueTx`, `avm.getTxStatus`, `platform.issueTx`, and `platform.getTxStatus`
- Matching server spans exist for those same requests
- Client and server spans for the same request share a trace ID and expected parent-child relationships
- A summary makes it easy to compare:
  - whole test duration
  - per-RPC duration
  - per-transaction issuance and confirmation duration

## Validation-First Approach

This work should be executed in implementation steps that each have an explicit automated validation loop.

Manual inspection is acceptable only during exploration, for example:

- confirming how current tracing configuration works
- checking what shape of trace attributes is most useful in Grafana
- comparing competing local stack options such as direct export versus a local collector

Once implementation starts, each step should produce a reusable automated check. "Run it once by hand and see if it works" is not a sufficient guard against regression.

For this plan, validation means one of:

- a unit test
- an integration test
- a tmpnet-focused test helper with an automated assertion
- an e2e test invocation that can be run repeatedly and checked programmatically

The final Banff proof of concept should therefore be backed by smaller automated tests for:

- local observability stack startup and readiness
- local Grafana and Tempo availability
- trace collector startup and readiness
- node-side trace export
- trace queryability
- tracing harness plumbing, following existing test conventions where possible
- trace propagation
- Banff end-to-end behavior

## Proposed Strategy

### Phase 1. Add tooling to start and verify a local observability stack

The first implementation step should be observability infrastructure, not Banff instrumentation.

#### Local versus remote monitoring

The existing monitoring workflow is remote-first: Prometheus runs in agent mode and pushes metrics to a remote Prometheus backend, and Promtail pushes logs to a remote Loki backend. Both require environment variables for URLs and credentials (`PROMETHEUS_*`, `LOKI_*`). This works well for CI and shared environments but does not support a self-contained local development workflow.

The local monitoring stack is an additional mode, not a replacement:

- **Remote mode (existing)**: `--start-metrics-collector` / `--start-logs-collector` with env vars for remote backends. Unchanged by this work.
- **Local mode (new)**: `--start-local-monitoring-stack` starts Grafana, Prometheus (server mode), Loki, and reconfigured collectors. No remote credentials required.

When local mode is active:

- Prometheus runs in server mode with local TSDB storage instead of agent mode with remote write
- Promtail pushes to the local Loki instead of a remote one
- Grafana is provisioned with datasources pointing at local Prometheus and Loki
- Test output emits Grafana links pointing at `http://127.0.0.1:3000` instead of the remote Grafana instance
- Collection checks (`--check-metrics-collected`, `--check-logs-collected`) query local backends

#### Process management versus Docker Compose

The local stack requires up to 5 services (Grafana, Prometheus, Loki, Promtail, and eventually Tempo). All are available as nix packages:

- `grafana` 12.3.0
- `prometheus` (already in flake)
- `grafana-loki` 3.6.3
- `promtail` (already in flake)
- `tempo` 2.9.0

Two management approaches are viable:

**Option A: Individual processes (current pattern)**

Extend the existing detached-process pattern in `monitor_processes.go`. Each service gets start/stop functions, PID files, readiness checks, and `tmpnetctl` commands. This is consistent with the existing Prometheus and Promtail management.

Advantages:
- No new dependency (no Docker required)
- Consistent with existing codebase conventions
- Each service independently startable/stoppable
- Works identically on all platforms with nix

Disadvantages:
- More code to manage 5 services
- No declarative dependency ordering between services

**Option B: Docker Compose**

Define a `docker-compose.yaml` with all local services. A single `tmpnetctl start-local-monitoring-stack` invokes `docker compose up -d`.

Advantages:
- Single file defines all services
- Built-in dependency ordering and health checks
- Familiar to most developers

Disadvantages:
- Adds Docker as a required dependency for local monitoring
- Differs from the existing nix-managed process pattern
- Configuration injection is less direct than writing config files

**Decision**: Use individual nix-managed processes (Option A) for consistency with the existing codebase. The number of services is manageable because they share common helpers (`startCollectorProcess`, `stopCollector`, `waitForReadiness`). `--start-local-monitoring-stack` composes these into a single entrypoint. If the process count grows beyond what's comfortable, Docker Compose can be reconsidered later.

## Trace Sink and Collection Plan

### 6. Treat trace collection as a first-class tmpnet collector

The existing tmpnet monitoring design is the right model to follow.

Today tmpnet already:

- starts background collectors for metrics and logs
- writes per-node service discovery configuration
- pushes data to remote backends
- emits Grafana links for viewing results

Relevant existing pieces:

- [tests/fixture/tmpnet/monitor_processes.go](/home/maru/src/avalanchego_maru-test-tracing/tests/fixture/tmpnet/monitor_processes.go)
- [tests/fixture/tmpnet/README.md](/home/maru/src/avalanchego_maru-test-tracing/tests/fixture/tmpnet/README.md)
- [tests/fixture/tmpnet/check_monitoring.go](/home/maru/src/avalanchego_maru-test-tracing/tests/fixture/tmpnet/check_monitoring.go)
- [tests/e2e/README.md](/home/maru/src/avalanchego_maru-test-tracing/tests/e2e/README.md)

Tracing should follow the same operational pattern:

- a tmpnet-managed trace collector process
- test and node exporters pointed at that collector or directly at the trace backend
- test output that includes a link to a GUI for immediate inspection

The word "sink" is reasonable informally, but the more precise terms here are:

- collector
- exporter destination
- backend

For this plan, "trace sink" means the backend traces are exported to and queried from.

### 7. Prefer a local Grafana + Tempo GUI workflow for the first pass

Given the repo already uses a `prometheus + loki + grafana` pattern, the most natural tracing analogue is:

- OpenTelemetry trace export
- Tempo as the trace backend
- Grafana as the viewer

Why this is the preferred direction:

- It matches the current monitoring architecture
- It supports end-to-end trace correlation well
- It preserves the "open a Grafana link and inspect immediately" workflow
- It avoids inventing a test-only trace UI
- It avoids depending on a remote trace backend for local development

For the Banff proof of concept, prefer this viewing experience:

- run the Banff test with tracing enabled
- tmpnet starts or confirms local tracing infrastructure is running
- traces are exported during test execution
- test logs emit a Grafana link filtered to the current network/test
- the resulting trace can be opened immediately in the browser

For the first pass, the tracing stack should run locally:

- local Tempo
- local Grafana
- optionally a local OpenTelemetry Collector between the test/node exporters and Tempo

All local services run as nix-provided binaries managed as detached processes, consistent with the existing Prometheus and Promtail pattern.

### 8. Start with tmpnet-managed collection rather than ad hoc local files

The earlier idea of a local file or in-memory artifact is still viable for debugging, but it does not satisfy the desired workflow well:

- it does not mirror existing tmpnet monitoring behavior
- it does not provide immediate GUI access
- it makes client/server correlation harder to inspect interactively

For this reason, the preferred first pass should be:

- local Tempo backend
- local Grafana UI
- a local tmpnet-managed collector process if needed

All managed as nix-provided binaries via the existing detached-process pattern. This is closer to how metrics and logs are already handled and should make the proof of concept more representative.

### 9. Trace export path: direct to Tempo, no intermediate collector

AvalancheGo already supports direct OTLP export via `--tracing-exporter-type=grpc --tracing-endpoint=localhost:4317`. Since Tempo can receive OTLP directly, no intermediate collector (OpenTelemetry Collector, Alloy, etc.) is needed for the local stack.

The trace export path for the PoC is:

- AvalancheGo nodes → OTLP gRPC → local Tempo (port 4317)
- Test harness (Phase 3) → OTLP gRPC → local Tempo (port 4317)

If an intermediate collector becomes necessary later (e.g., for batching, attribute injection, or routing to multiple backends), it can be added without changing the node or test configuration — just point the exporters at the collector instead of Tempo.

### 9.1 Add a post-run trace collection check parallel to metrics and logs

If tracing is enabled, shutdown should verify that traces were actually collected, just as the current harness can verify that metrics and logs were collected. Setup traces should count for this check because tmpnet setup is always part of the run and should always be traced when tracing is enabled.

Suggested shape:

- add `tmpnet.CheckTracesExist(...)`
- add a corresponding e2e flag such as `--check-traces-collected`
- register a deferred cleanup in the e2e harness when tracing is enabled and trace checking is requested

This check should query the trace backend or collector for evidence that traces for the current network UUID were received.

The goal is to turn "tracing was enabled" into "traces were actually exported and queryable."

### 9.2 Add reuse-network tracing compatibility checks

Running against an existing network creates a configuration consistency problem that should be handled explicitly.

If a test run requests tracing and `--reuse-network` or `--restart-network` is used, the harness should inspect the reused network configuration and verify whether tracing was enabled when that network was created.

Expected behavior:

- if the run requests tracing and the reused network is not configured for tracing:
  - error by default
  - warn-only behavior should require a deliberate opt-out flag if it is supported at all later
- if the run does not request tracing but the reused network has tracing enabled:
  - warn, because the run configuration and network configuration are inconsistent

This check should happen before tests begin, so the operator gets a fast and clear failure instead of silently running with missing spans.

The reused network inspection should be based on persisted network/runtime configuration rather than inferred behavior.

### 10. Extend the nix flake to provide local monitoring stack tools

The current development shell already provides:

- `prometheus`

### 15. Propagate trace context from client to server

This is the critical requirement for end-to-end visibility.

Without propagation:

- client and server spans exist but are disconnected

With propagation:

- a client RPC span becomes the parent of the server request span
- the full path from test to server handler is visible in one trace

Suggested work:

- Use standard trace propagation headers on the client side
- Ensure server tracing extracts those headers before creating request spans
- Validate the resulting spans share trace IDs and have the expected parent-child relationships

## Risks and Open Questions

### Resolved

- **Local service management**: Nix-managed detached processes, not Docker. All required packages are available in nixpkgs. (See Phase 1 strategy for rationale.)
- **AvalancheGo tracing support**: Already has full OTLP export (gRPC and HTTP), node-level flags, and server-side API handler tracing. No additional plumbing needed for node-side export.
- **Trace collector**: For the PoC, prefer direct OTLP export from nodes to local Tempo without an intermediate collector. AvalancheGo already supports this via `--tracing-exporter-type=grpc --tracing-endpoint=localhost:4317`.
- **Flag interaction**: `--start-local-monitoring-stack` subsumes `--start-metrics-collector` and `--start-logs-collector`. The local stack starts its own Prometheus (server mode) and Promtail (local-mode), so the remote-push collector flags are redundant. If both are specified, the local stack wins and the remote collector flags are ignored.

### Open

- The existing server tracing path may assume node-level configuration that is not yet easy to control from tmpnet.
- The current client request stack may not expose enough hooks for propagation without a small `utils/rpc` refactor.
- Polling-heavy flows such as `getTxStatus` may generate many spans; the first pass may need filtering or summarization to keep output readable.
- The Banff test and tmpnet setup will generate startup traffic before the main test actions; reporting should separate setup spans from test-action spans rather than trying to omit setup from tracing.
